Assignment Overview
3
What is A3?
The final assignment after this, you have
built a simple simulated OS!
In A2, programs had to fully fit in shell
memory. A3 removes this limitation using
demand paging.
Programs are split into pages (3 lines each).
Only needed pages are loaded. Old pages
are swapped out when memory is full.
Key Constraints
• Only RR policy tested (time slice = 2)
• No multithreading, no background (#)
• Same program name allowed in exec
(process sharing)
• Frame store size always a multiple of
3
• Terminated processes do NOT clean
up their frames
• Both source and exec must use paging
What’s New
• Shell memory split into frame store (code)
+ variable store
• PCBs have page tables instead of
contiguous line_base / line_count
• Scripts copied to backing_store/ directory
for disk-backed loading
• Page faults handled with LRU eviction
when frame store is full
• Duplicate script names now allowed in
exec. Compile-time flags set memory sizes.
What Stayed the Same
• Scheduling queue logic
• FCFS / SJF / RR / AGING policies
• Shell parseInput (semicolons etc.)
• runSchedule loop: dequeue → run → re-
enqueue
• All A1/A2 shell commands: help, quit, set,
print, echo, my_ls, etc.

Key Points
• Each page = 3 lines of code
• Each frame = 3 contiguous slots in the
frame store
• Frame store and variable store are
separate (or logically partitioned)
• Sizes set at compile time with -D flags
(more on this in Task 2)
• Pages are NOT loaded contiguously —
page tables handle the mapping
Address translation:
page = PC / 3, offset = PC % 3
physical index = page_table[page] + offset

Task 1 : Paging Infrastructure
5
Page Tables & Process Sharing
Page Table Example
Page # Frame Start Index Status
0 0 Loaded
1 6 Loaded
2 -1 Not loaded
• Each PCB has its own page table array
• page_table[i] = frame start index, or -1 if not
loaded
• Size = ceil(total_lines / 3)
Process Sharing
exec prog1 prog1 RR is now legal!
When two processes share the same
script name:
• They share the same loaded code
pages in memory
• Each has its own PCB and PC
• If one process loads a page, the
other can use it (no page fault)
• Page table entries must be kept in
sync
• If a shared page is evicted, update
ALL page tables that reference it

Compilation

Startup message (replaces version string):

Initial Code Loading Rules
• Only load the first 2 pages of each program at start
• If program < 3 lines, only 1 page is loaded
• Remaining pages loaded on demand (page faults)
• Programs start executing per scheduling policy (RR, time slice = 2)
Page Fault Handling
1. Process tries to execute
instruction from an unloaded page → 2. Process is interrupted and
placed at back of ready queue → 3. Missing page is loaded from the
backing file into frame store
If frame store has free space:
• Allocate a free frame
• Print: "Page fault!"
• Load the page, update page table
If frame store is FULL:
• Pick a victim frame (LRU — Task 3)
• Print victim page contents
• Evict it, load new page, update ALL page
tables
Task 3 : LRU Replacement Policy
7
Least Recently Used: evict the frame that was accessed least recently across ALL processes
Requirements
• Must be accurate LRU, not an
approximation
• Track access times for every frame in the
frame store
• Update the timestamp every time a
frame is accessed (i.e., when an
instruction from that frame is executed)
• On eviction, select the frame with the
smallest (oldest) timestamp
• A page fault from P1 can evict a page
from P2
Implementation Hints
• Option A: Linked list with move-to-front
on access (classic approach)
• Option B: Timestamp counter per frame
(simpler to implement)
Whichever you choose, you need a way to:
• Mark a frame as accessed
• Find the LRU frame quickly
• Map frames back to page table
entries for invalidation
Remember: When you evict a frame, you must invalidate ALL page table entries (across all
PCBs) that point to that frame.

Tips & Common Pitfalls
10
✓ Do’s
• Start with Task 1 : get your A2 tests
passing with pages before adding
demand paging
• Test incrementally: TC1/TC2 first (no
faults), then TC3 (faults), then TC4/TC5
• Use your A2 solution as the base
(recommended) or the official A2
solution
• Keep your backing store cleanup reliable
(rm -rf backing_store)
• Re-compile between test cases
(different framesize!)
• Redownload your submission zip and
verify it compiles from scratch
✗ Don’ts / Watch Out
• Don’t forget to update ALL page tables
on eviction (shared processes!)
• Don’t clean up frames when a process
terminates (spec says don’t)
• Don’t assume pages load contiguously
that’s the whole point of page tables
• Don’t use approximate LRU (e.g., clock
algorithm) must be accurate
• Don’t hardcode outputs: automatic
zero
• Watch for off-by-one errors in page
table indexing (PC / 3 vs PC % 3)
